# Yang 抢课脚本 v1.2.1：前台页面优先设计

日期：2026-07-13

## 目标

解决 v1.2.0 的两个实际问题：Test/立即启动在创建后台 Worker 和服务器校时期间响应慢；用户手动打开的 ME/FE 详情页因缺少 Worker 标记而只能观察，不能参与自动选课。

本次采用“前台页面优先、专用 Worker 兜底”的混合架构。抢课开关仍由全局运行状态控制；`STOPPED` 时只识别，绝不提交动作。

## 已确认根因

1. `startImmediate()` 在发布 `RUNNING` 前等待一次没有超时的 BNBU `HEAD` 校时请求。请求慢或悬挂时，所有自动动作都会被推迟。
2. `test()`、`startImmediate()` 和 `startScheduled()` 都等待 `ensureWorkers()`。旧实现为每个目标依次读取、写入、复核 GM Worker 池，再调用 `GM_openInTab`，所以按钮响应受三组跨标签存储操作和后台标签加载影响。
3. 未带 `yang-worker` Hash 的详情页无法构造 `workerSlot`，`claimCurrentWorker()` 必然失败，`scan()` 返回 `inactive-worker`。这是代码明确设定的行为，不是页面识别失败。

## 运行角色

### 控制页

状态页继续承载完整控制面板、课程配置、即时启动、预约启动和 Stop。控制页本身不解析详情表格，也不提交课程动作。

### 前台优先页（Hot Page）

任何由用户手动打开、没有 Worker 标记的 `eleDetail.do` 页面都成为前台优先页：

- 根据页面表格识别本页类别，而不是依赖 URL 参数。
- 一次扫描本类别的全部目标，例如一个 ME 页面同时检查 COMP3073 和 COMP4213。
- 页面初始化后立即扫描，不等待校时、Worker 池或下一次轮询。
- 全局为 `RUNNING` 时允许进入共享动作队列；`STOPPED` 或预约尚未开始时仅显示识别结果。
- 使用与专用 Worker 相同的动作白名单、精确课程匹配、原生确认桥、共享动作锁和提交结果验证。
- 页面关闭时释放本地计时器；不会清空全局运行状态。

### 专用 Worker

保留 ME-1、ME-2、FE-1 三个专用 Worker。它们只作为缺页、页面关闭、页面跳转或前台页未覆盖目标时的兜底路径。

- Worker 仍按目标独立刷新。
- 同一目标被前台页和 Worker 同时发现时，共享状态锁、动作签名去重和 pending action 保证只提交一次。
- 前台页候选在尚未取得动作锁时排在同目标后台候选之前；已开始的合法提交不被强行中断。

## 即时路径

`立即启动`按以下顺序执行：

1. 立即发布全局 `RUNNING`。
2. 已打开的前台优先页通过控制状态监听立刻扫描并提交。
3. 当前页面若本身是详情页，也立即扫描。
4. 异步校准服务器时间，不阻塞步骤 1–3。
5. 异步批量预留并补建缺失的专用 Worker。

因此，已经打开并显示 Select 或 Join Waiting List 的页面不再等待新标签页。

## Test 与预约启动

### Test

- 立即扫描当前页面并更新 UI，不执行动作。
- 异步预热缺失 Worker；按钮返回不等待 Worker 完整加载。
- 后续 Worker 上线后通过共享状态回传各目标识别结果。

### 预约启动

- 先发布预约状态并异步预热 Worker。
- 使用最近一次有效校时；后台发起新的限时校时并更新偏移。
- 进入预约窗口后先切换 `RUNNING`，再执行扫描，不等待新的网络请求。

## 校时与 Worker 创建优化

- BNBU 校时请求增加 1 秒硬超时；失败时记录原因并使用本机时间，后续定期重试。
- 即时启动永远不等待校时。
- Worker 预留改为一次读取、一次批量写入，然后连续调用 `GM_openInTab`，删除逐目标的读写复核往返。
- Opening Lease 延长到 60 秒，连续点击 Test/启动不重复开页；到期且没有心跳时只补建缺失槽位。

## 状态与 UI

手动详情页使用只读迷你条，标题显示“Yang 前台优先页”，并显示：

- 当前类别与覆盖目标；
- `仅识别 / 高速扫描 / 正在提交 / 已加入轮候 / 已抢到 / 执行失败`；
- 最近扫描时间和失败原因。

控制中心沿用每门课程状态卡。状态来源可标记为“前台页”或具体 Worker，便于判断实际执行路径。

## 错误处理

- 页面类别未知、目标重复匹配、函数不存在或确认文本不符时禁止提交并显示原因。
- 校时失败不阻塞即时动作，只影响预约时间精度提示。
- 前台页提交后导航时由 session pending action 校验结果；未成功按已有退避规则恢复扫描。
- Stop 立即取消前台页和 Worker 的刷新、MutationObserver、动作队列与自动确认。

## 测试边界

新增失败测试后再实现：

1. 未标记详情页在 `RUNNING` 时可扫描并执行 Select/Join Waiting List。
2. 未标记详情页在 `STOPPED` 时只识别、不执行。
3. 一个 ME 前台页同时产生两个目标评估。
4. `startImmediate()` 在校时 Promise 悬挂时仍先进入 `RUNNING` 并触发前台页动作。
5. 校时在 1 秒后回退本机时间。
6. Test 在 Worker 尚未加载时立即返回本地扫描结果。
7. 三个 Worker 使用一次批量预留，反复 Test/Start 不重复打开。
8. 前台页与专用 Worker 同时发现同一目标时只提交一次。
9. 原有 Select from Waiting、250 ms 共享队列、预约窗口、Stop 和禁止动作测试继续通过。

## 非目标

- 不调用隐藏选课 API。
- 不自动登录或处理验证码。
- 不执行 Drop、Replace 或 Exit Waiting。
- 不取消三个专用 Worker 的兜底架构。
- 不让 `STOPPED` 页面自动提交任何课程动作。
